## 1. Project Overview

**Quante** is an AI-native, white-label e-commerce builder SaaS. Users describe the store they want, the in-product AI (called "Quante", powered by the Claude API) generates a complete Next.js storefront, users iterate conversationally, and can then export the full source as a downloadable, deploy-ready ZIP or deploy it directly to Vercel with one click.

### Two completely separate surfaces

| Surface | Description | Design |
|---|---|---|
| **The Quante platform** | The builder/SaaS app users log into | Dark, technical, premium — Linear/Vercel aesthetic |
| **Generated storefronts** | White-label shops Quante produces | Brand-appropriate per brief, zero Quante branding |

**Critical non-negotiables:**
- White-label output — generated shops carry no Quante branding/telemetry
- Portable code — every export is a plain Next.js project running with `npm install && npm run dev`
- Manifest/code-gen driven — AI produces structured output that slots into a deterministic scaffold

### Tech Stack

**Platform:**
- Next.js 16.2.7 (App Router) + TypeScript 5
- Tailwind CSS v4 + `tw-animate-css` + `shadcn/ui`
- Framer Motion 12
- Clerk (auth — replaced Supabase Auth)
- Supabase (Postgres, RLS, Storage)
- Stripe (credit packs, Agency subscription, hosting subscription, Connect)
- Claude API (`@anthropic-ai/sdk 0.100.1`) — server-side only
- `jszip` for ZIP export
- `@vercel/sdk` for Vercel API

**Generated storefronts (scaffold):**
- Next.js 16.2.7 + TypeScript
- Tailwind CSS v4
- Framer Motion, lucide-react
- Cart state via React Context (localStorage-backed)
- CSS custom properties for theming (`--color-bg`, `--color-text`, etc.)
- No Stripe keys baked in — merchant provides their own via Vercel env vars or Quante managed payments

## 4. Database Tables

> All server-side code uses `supabaseAdmin` (service role), which bypasses RLS. RLS is a defence-in-depth layer only.
> Auth: Clerk JWT. RLS policies use `auth.jwt() ->> 'sub'` (not `auth.uid()`) since Clerk user_id is a text string, not a UUID.

       │   streams chunk events to client (live terminal display)
       ├── parse <file path="...">...</file> blocks
       ├── create projects row in DB

       ├── ensureVercelProject(slug)
       ├── buildStoreFiles(codeFiles) → scaffold + AI files merged

       │   assigns subdomain: slug.HOSTING_ROOT_DOMAIN

       └── send { type: 'done', projectId, versionId, deploymentId, previewUrl }

5. User iterates via chat (1 cr each)

       ├── load current code_versions.files

       ├── parse <reply> + <file> blocks

### Tiers
- **free/credit**: Credit-based. Rate limits: generate 5/hr, iterate 60/hr, section 15/hr. Project limit: 3.

### Ledger mechanics
- `credit_ledger` is append-only: `{ delta, reason, ref_id, balance_after }`
- Current balance = `balance_after` of the most recent row
- Debit happens AFTER successful work. Exception: `generate` debits after saving code_version but before deploy.
- Welcome grant: automatically inserted on first `/api/credits/balance` call (or first dashboard visit via `ensureWelcomeGrant`)
- Agency users' iterate/fix skip credit debit entirely

### Credit packs (Stripe)
- 100 cr = €9.99
- 300 cr = €24.99 (popular)
- 1000 cr = €69.99
- Stripe webhook (`checkout.session.completed` with no `type` metadata) → inserts purchase row + credits ledger entry

## 11. Vercel Integration (`lib/hosting/vercel.ts`)

Uses `@vercel/sdk` + `VERCEL_TOKEN` + `VERCEL_TEAM_ID`.

### `ensureVercelProject(slug)`
Search Vercel for project by name (exact match). If not found, create with `framework: 'nextjs'`. Returns `{ vercelProjectId }`. Idempotent.

### `createPreviewDeployment(vercelProjectId, files, storeSlug?)`
Sends all files to Vercel deployments API (`target: 'production'`). If `storeSlug` provided, calls `attachDomain(vercelProjectId, slug.HOSTING_ROOT_DOMAIN)` — attaches the subdomain. Returns `{ deploymentId, url }` where `url` is either the subdomain URL or the raw Vercel URL.

### `createDeployment(vercelProjectId, files, { target: 'production' })`
Same as above without subdomain assignment. Used for live deploys where domain is assigned later via `attachDomain` in the GET `/api/deploy` handler.

### `streamDeploymentLogs(deploymentId, onEvent, signal?)`
Calls Vercel events API (`/v2/deployments/{id}/events?direction=forward&follow=1`). Parses NDJSON. Emits events via callback. Detects `readyState === 'READY'` or `'ERROR'` in payload to emit terminal events and return.

### `getDeploymentStatus(deploymentId)`
Maps Vercel `readyState` → `'queued' | 'building' | 'ready' | 'error' | 'canceled'`.

### `getBuildError(deploymentId)`
Fetches last 20 stderr lines from Vercel events API.

### `attachDomain(vercelProjectId, domain)`
Calls Vercel add-domain API. Returns `{ verified: boolean, dnsInstructions? }`. Idempotent (Vercel returns 400 if already attached, which is caught and ignored).

## 12. Known Bugs / Gotchas

### maxDuration Limits (Vercel)
- `/api/quante/generate`: `maxDuration = 300` (5 min). Claude generation with claude-opus-4-7 at 64k tokens can approach this. If stream ends without `done` event, the `/new` page shows a graceful timeout message ("check dashboard").
- `/api/quante/iterate`: `maxDuration = 300`. With large stores (many files), sending all files as context can be slow.
- `/api/deploy/logs`: `maxDuration = 300`. Build logs stream for the duration of the Vercel build (~2-3 min typically).

### RLS auth.uid() vs auth.jwt() ->> 'sub'
The project switched from Supabase Auth to Clerk. Old policies used `auth.uid()`. New policies must use `(auth.jwt() ->> 'sub')`. Some migrations still use `auth.uid()::text` in store-related policies (store_orders, store_earnings, etc.) — this works because Supabase maps the JWT `sub` to `auth.uid()` when a JWT is set. However, when `supabaseAdmin` (service role) is used, RLS is bypassed entirely. Be careful if any code uses the anon-key client — the JWT needs to be a valid Clerk JWT passed as the auth header.

### supabase/server.ts Re-exports Admin Client
`createClient()` in `lib/supabase/server.ts` just returns `supabaseAdmin`. There is NO per-request cookie-based client. This means all queries run as service role (RLS bypassed). Filtering must be done explicitly by `userId`. This is intentional but means client-side leaks are impossible only because no client code has Supabase credentials.

### Lucide Icon Names
`lucide-react ^1.17.0` has a different icon set than v0.x. Social icons (Instagram, Twitter, etc.) do not exist. `sanitizeLucideImports()` in `build.ts` replaces them automatically. Always check SYSTEM_PROMPT_CODE_GENERATION for the current allowlist.

### Middle Dot in Footer (Unicode Turbopack Issue)
Previous bug: a middle dot character (·) in the Footer template caused Turbopack to fail. Fixed by replacing with a dash (-). See commit `469e589`.

### Code Versions vs Manifest Versions
The project has TWO generation approaches:
- **Legacy (manifest)**: Claude generates a `ShopManifest` JSON → renderer maps it to components. Uses `manifest_versions` table. Still used for the preview iframe in `app/(preview)/`.
- **Current (code-gen)**: Claude generates actual TypeScript/React files. Uses `code_versions` table. Used by all new generate/iterate/fix/deploy/export routes.

Export route (`/api/export`) checks `code_versions` first, falls back to `manifest_versions`.

### Token Limits
`AGENCY_TOKEN_CAP = 64_000` for agency users. Free users also use 64,000 in iterate. Generation uses 64,000. These are `max_tokens` values — the actual response may be shorter. If output is truncated (no closing `</file>` tag), parsing fails and returns error.

### Deploy Credits Timing
Deploy credits (5) are debited AFTER the deployment is confirmed ready (in `GET /api/deploy?id=...`). If the user never polls (closes browser), the deploy still happens on Vercel but credits are not debited. This is intentional (user-friendly) but means a free deployment is possible if the poll never happens.

### Agency Project Archiving
On Agency plan cancellation webhook, projects beyond the free limit (3) are set to `status='archived'`. They're hidden but not deleted. User sees a warning to reactivate. If they upgrade again, archives are NOT automatically restored — they need to be unarchived manually.

### Stripe webhook: `type` metadata
The Stripe webhook (`/api/stripe/webhook`) distinguishes events by `session.metadata.type`:
- Missing/undefined type + credits metadata = credit pack purchase
- `type = 'store_sale'` = deployed store checkout
- `type = 'hosting'` = hosting subscription (subscription webhook handles separately)
- `type = 'domain_purchase'` = domain registration flow

### preview vs production deployment distinction
In `code_versions`/`deployments`, preview deployments have `domain = null`. Live (production) deployments have `domain = 'slug.quante.app'`. StudioClient uses this to distinguish: if latest deployment has a `domain`, it's a live deploy; otherwise it's a preview.

- **Phase 2 (Generate + Studio)**: Full generation pipeline via code-gen mode (claude-opus-4-7). Studio with chat, preview iframe, log streaming, version history.
- **Phase 3 (Iterate + Versioning)**: Iterate works, version history shown in Studio, restore older version works.
- **Phase 4 (Credits + Stripe)**: Credit ledger, atomic debit after success, packs, Stripe webhook, Agency subscription, hosting subscription.
- **Phase 5 (Export)**: ZIP export in both modes (code-gen + legacy manifest). Agency white-label scrub working. Test in `__tests__/export-whitelabel.test.mjs`.
- **Deploy to Vercel**: One-click deploy from Studio, SSE log streaming, auto-fix build errors (up to 3 attempts), 30-day trial, credit debit on success.
- **Czech payment gateways**: Comgate, GoPay, PayPal IPN webhooks.
- **Managed payments**: Quante handles Stripe for deployed stores (QUANTE_API_KEY auth), records earnings, sends customer+merchant emails via Resend.
- **Shipping**: Zásilkovna (Packeta CZ + international), DHL Express integration.
- **Inventory**: `store_inventory` table, atomic stock decrement, low-stock view.
- **Invoice generation**: PDF invoices via `/invoice/[orderId]`.
- **Domain management**: Search (Namecheap), purchase (Stripe → Namecheap), attach to Vercel, DNS verification.
- **Admin panel in Studio**: Orders (Stripe + store), customers, revenue chart, shipment creation, payout account, IBAN management.
- **Landing page** (`/`): Cinematic scroll with sticky sections, horizontal showcase, pricing preview.
- **Pricing page** (`/pricing`): Full pricing info, credit packs, hosting details, Agency plan, FAQ.
- **Agency tier**: Batch generation limit (20), no credit cost, subscription management, project archiving on downgrade.
- **Vision analysis**: Image → brand palette/typography detection.
- **Unsplash image suggestions**: Product photo finder.
- **Stripe Connect**: Per-project connected accounts for direct merchant payouts.
- **Custom domain**: Connect user-owned domains to deployed stores.

### In Progress / Known Issues
- **Preview iframe**: The preview uses `/preview/[id]/...` routes inside the platform for legacy manifest-based stores. For code-gen stores, the preview is the actual Vercel preview deployment URL (loaded in iframe). The legacy preview components still exist but code-gen stores bypass them.
- **maxDuration on Vercel**: 300s routes are at the Vercel max for Pro plans. Long briefs with claude-opus-4-7 can approach this. No graceful timeout/resume mechanism — user sees timeout error message.
- **RLS inconsistency**: Some older migration policies use `auth.uid()::text` instead of `(auth.jwt() ->> 'sub')`. This works because Supabase maps the JWT sub claim to uid, but if the JWT structure changes it could break. All server code uses service-role client anyway.
- **Admin panel feature flag**: `adminPanel?: boolean` exists in ShopManifest but the admin panel in the Studio is always available (not gated by the flag in the current UI).

### Not Yet Implemented
- **Phase 6 (Polish)**: showcase page content, full docs page, empty/error states, rate limit UI.
- **Supabase anon-key client**: `lib/supabase/client.ts` exists but is not used — all auth is via Clerk + service role.
- **Middleware**: `lib/supabase/middleware.ts` exists but route protection may be minimal.
- **`/showcase` page**: Route exists but content is likely placeholder.
- **Three.js hero**: Mentioned in CLAUDE.md but not implemented.
