## 1. Project Overview

**Quante** is an AI-native, white-label e-commerce builder SaaS. Users describe the store they want, the in-product AI (called "Quante", powered by the Claude API) generates a complete Next.js storefront, users iterate conversationally, and can then export the full source as a downloadable, deploy-ready ZIP or deploy it directly to Vercel with one click.

### Two completely separate surfaces

| Surface | Description | Design |
|---|---|---|
| **The Quante platform** | The builder/SaaS app users log into | Dark, technical, premium вЂ” Linear/Vercel aesthetic |
| **Generated storefronts** | White-label shops Quante produces | Brand-appropriate per brief, zero Quante branding |

**Critical non-negotiables:**
- White-label output вЂ” generated shops carry no Quante branding/telemetry
- Portable code вЂ” every export is a plain Next.js project running with `npm install && npm run dev`
- Manifest/code-gen driven вЂ” AI produces structured output that slots into a deterministic scaffold

### Tech Stack

**Platform:**
- Next.js 16.2.7 (App Router) + TypeScript 5
- Tailwind CSS v4 + `tw-animate-css` + `shadcn/ui`
- Framer Motion 12
- Clerk (auth вЂ” replaced Supabase Auth)
- Supabase (Postgres, RLS, Storage)
- Stripe (credit packs, Agency subscription, hosting subscription, Connect)
- Claude API (`@anthropic-ai/sdk 0.100.1`) вЂ” server-side only
- `jszip` for ZIP export
- `@vercel/sdk` for Vercel API

**Generated storefronts (scaffold):**
- Next.js 16.2.7 + TypeScript
- Tailwind CSS v4
- Framer Motion, lucide-react
- Cart state via React Context (localStorage-backed)
- CSS custom properties for theming (`--color-bg`, `--color-text`, etc.)
- No Stripe keys baked in вЂ” merchant provides their own via Vercel env vars or Quante managed payments

## 9. Credit System

### Costs (`lib/config.ts`)
| Action | Credits |
|---|---|
| Generate (full store) | 10 |
| Iterate (chat message) | 1 |
| Section regenerate | 2 |
| Custom component | 3 |
| Export ZIP | 5 |
| Export ZIP + admin panel | 10 |
| Deploy to hosting | 5 |
| Auto-fix build error | 2 |
| Vision image analysis | 1 |
| Welcome grant (on signup/first balance check) | +25 |

### Tiers
- **free/credit**: Credit-based. Rate limits: generate 5/hr, iterate 60/hr, section 15/hr. Project limit: 3.
- **agency**: No credit cost for iterate/fix. Unlimited generations. Project limit: 20. Rate limit: 10 calls/min on code_versions.

### Ledger mechanics
- `credit_ledger` is append-only: `{ delta, reason, ref_id, balance_after }`
- Current balance = `balance_after` of the most recent row
- Debit happens AFTER successful work. Exception: `generate` debits after saving code_version but before deploy.
- Welcome grant: automatically inserted on first `/api/credits/balance` call (or first dashboard visit via `ensureWelcomeGrant`)
- Agency users' iterate/fix skip credit debit entirely

### Credit packs (Stripe)
- 100 cr = в‚¬9.99
- 300 cr = в‚¬24.99 (popular)
- 1000 cr = в‚¬69.99
- Stripe webhook (`checkout.session.completed` with no `type` metadata) в†’ inserts purchase row + credits ledger entry

## 11. Vercel Integration (`lib/hosting/vercel.ts`)

Uses `@vercel/sdk` + `VERCEL_TOKEN` + `VERCEL_TEAM_ID`.

### `ensureVercelProject(slug)`
Search Vercel for project by name (exact match). If not found, create with `framework: 'nextjs'`. Returns `{ vercelProjectId }`. Idempotent.

### `createPreviewDeployment(vercelProjectId, files, storeSlug?)`
Sends all files to Vercel deployments API (`target: 'production'`). If `storeSlug` provided, calls `attachDomain(vercelProjectId, slug.HOSTING_ROOT_DOMAIN)` then **always returns `https://slug.HOSTING_ROOT_DOMAIN`** regardless of whether `attachDomain` throws (domain is almost certainly already attached from a prior deploy; all non-fatal errors are logged and swallowed). If `storeSlug` is absent, falls back to raw Vercel URL.

### `createDeployment(vercelProjectId, files, { target: 'production' })`
Same as above without subdomain assignment. Used for live deploys where domain is assigned later via `attachDomain` in the GET `/api/deploy` handler.

### `streamDeploymentLogs(deploymentId, onEvent, signal?)`
Calls Vercel events API (`/v2/deployments/{id}/events?direction=forward&follow=1`). Parses NDJSON. Emits events via callback. Detects `readyState === 'READY'` or `'ERROR'` in payload to emit terminal events and return.

### `getDeploymentStatus(deploymentId)`
Maps Vercel `readyState` в†’ `'queued' | 'building' | 'ready' | 'error' | 'canceled'`.

### `getBuildError(deploymentId)`
Fetches last 20 stderr lines from Vercel events API.

### `attachDomain(vercelProjectId, domain)`
Calls Vercel add-domain API. Returns `{ verified: boolean, dnsInstructions? }`. Idempotent (Vercel returns 400 if already attached, which is caught and ignored).

## 12. Known Bugs / Gotchas

### maxDuration Limits (Vercel)
- `/api/quante/generate`: `maxDuration = 300` (5 min). Claude generation with claude-opus-4-7 at 64k tokens can approach this. If stream ends without `done` event, the `/new` page shows a graceful timeout message ("check dashboard").
- `/api/quante/iterate`: `maxDuration = 300`. With large stores (many files), sending all files as context can be slow.
- `/api/deploy/logs`: `maxDuration = 300`. Build logs stream for the duration of the Vercel build (~2-3 min typically).

### RLS auth.uid() vs auth.jwt() ->> 'sub'
The project switched from Supabase Auth to Clerk. Old policies used `auth.uid()`. New policies must use `(auth.jwt() ->> 'sub')`. Some migrations still use `auth.uid()::text` in store-related policies (store_orders, store_earnings, etc.) вЂ” this works because Supabase maps the JWT `sub` to `auth.uid()` when a JWT is set. However, when `supabaseAdmin` (service role) is used, RLS is bypassed entirely. Be careful if any code uses the anon-key client вЂ” the JWT needs to be a valid Clerk JWT passed as the auth header.

### supabase/server.ts Re-exports Admin Client
`createClient()` in `lib/supabase/server.ts` just returns `supabaseAdmin`. There is NO per-request cookie-based client. This means all queries run as service role (RLS bypassed). Filtering must be done explicitly by `userId`. This is intentional but means client-side leaks are impossible only because no client code has Supabase credentials.

### Lucide Icon Names
`lucide-react ^1.17.0` has a different icon set than v0.x. Social icons (Instagram, Twitter, etc.) do not exist. `sanitizeLucideImports()` in `build.ts` replaces them automatically. Always check SYSTEM_PROMPT_CODE_GENERATION for the current allowlist.

### Middle Dot in Footer (Unicode Turbopack Issue)
Previous bug: a middle dot character (В·) in the Footer template caused Turbopack to fail. Fixed by replacing with a dash (-). See commit `469e589`.

### Code Versions vs Manifest Versions
The project has TWO generation approaches:
- **Legacy (manifest)**: Claude generates a `ShopManifest` JSON в†’ renderer maps it to components. Uses `manifest_versions` table. Still used for the preview iframe in `app/(preview)/`.
- **Current (code-gen)**: Claude generates actual TypeScript/React files. Uses `code_versions` table. Used by all new generate/iterate/fix/deploy/export routes.

Export route (`/api/export`) checks `code_versions` first, falls back to `manifest_versions`.

### Token Limits
`AGENCY_TOKEN_CAP = 64_000` for agency users. Free users also use 64,000 in iterate. Generation uses 64,000. These are `max_tokens` values вЂ” the actual response may be shorter. If output is truncated (no closing `</file>` tag), parsing fails and returns error.

### Deploy Credits Timing
Deploy credits (5) are debited AFTER the deployment is confirmed ready (in `GET /api/deploy?id=...`). If the user never polls (closes browser), the deploy still happens on Vercel but credits are not debited. This is intentional (user-friendly) but means a free deployment is possible if the poll never happens.

### Agency Project Archiving
On Agency plan cancellation webhook, projects beyond the free limit (3) are set to `status='archived'`. They're hidden but not deleted. User sees a warning to reactivate. If they upgrade again, archives are NOT automatically restored вЂ” they need to be unarchived manually.

### Stripe webhook: `type` metadata
The Stripe webhook (`/api/stripe/webhook`) distinguishes events by `session.metadata.type`:
- Missing/undefined type + credits metadata = credit pack purchase
- `type = 'store_sale'` = deployed store checkout
- `type = 'hosting'` = hosting subscription (subscription webhook handles separately)
- `type = 'domain_purchase'` = domain registration flow

### Preview URL вЂ” always use storeUrl, never raw vercel.app URLs
Raw Vercel deployment URLs (e.g. `axiom-az20nza2a-mikelfxs-project.vercel.app`) are blocked in iframes by Vercel's security headers. The canonical `storeUrl` (computed server-side) is ALWAYS used instead. The `resolveUrl()` helper in StudioClient swaps any `vercel.app` URL for `storeUrl` at every `setPreviewUrl()` call site.

### Preview vs production deployment distinction
In `deployments` table: iterate/fix/auto-deploys have `domain = null`. Live (production) deploys have `domain = 'slug.stores.quantecode.com'`. StudioClient uses `!!d.domain` (`isLiveDeploy`) to distinguish:
- Live deploy building в†’ shows Deploy tab loading card, polls `GET /api/deploy?id=...` every 12s
- Auto-deploy building в†’ polls status once on page load (to avoid SSE triggering auto-fix for stale errors), then starts `startLogStreaming` only if genuinely still building
- Live deploy ready в†’ updates `deployStatus`, `deployUrl`, `deployDomain`
- Auto-deploy ready в†’ updates `previewUrl = storeUrl`, `previewReady = true`

### SSE log streaming вЂ” termination guarantee
`streamDeploymentLogs` can return without emitting a terminal event (build was already complete when streaming started). `GET /api/deploy/logs` handles this: after `streamDeploymentLogs` returns, if no terminal event was emitted, it calls `getDeploymentStatus` and sends appropriate `stream_end` + closes the SSE controller. This prevents the client from hanging indefinitely on "Building on VercelвЂ¦".

### First-time generate requires Push to Live
After `POST /api/quante/generate`, the client sets `showPushToLive = true` instead of starting log streaming. The auto-deploy runs on Vercel in the background but the client shows a "Push to Live вЂ” 5 cr" button. The user must click it to trigger a proper production deploy that attaches the subdomain. Iterate/fix auto-deploys (not first generation) still use the existing log streaming + `previewReady` flow.

- **Auto-deploy on first generate is wasted**: The generate API creates a Vercel deployment (auto-deploy), but StudioClient now shows "Push to Live" instead of tracking it. A second production deploy is triggered when the user clicks "Push to Live". The first deploy runs silently and is effectively unused.
